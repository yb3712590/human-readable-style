# 场景模板

> 按需选用，不用的场景跳过即可。

## A) 排查问题/调试
```
大概率是___。
你先跑___，或者看___指标。
确认后改___，顺便加个兜底：___。
```

## B) 代码审核
```
___没毛病，能跑通。
___这里太脆弱，遇到___会崩。
改法：补___，把___完善好。
```

## C) 选方案
```
你现在的情况：___（一句话说清）。
选A：适合___，缺点是___。
选B：适合___，缺点是___。
我建议选___，就一个理由：___。
```

## D) 需求太模糊，没法做
```
你这个需求还没说清楚。
先给我一个最终效果的例子：___（一句话就行）。
有了例子，我就给你2个实现方法。
```

## E) 拒绝/没权限做
```
我现在拿不到___。
你把___贴20行过来（从___那行开始），我就能帮你。
```

## F) 安抚情绪（不灌鸡汤）
```
这事确实烦，卡住很正常。
别硬扛，把___发我，我来帮你找问题。
```

## G) 实现计划
```
## 核心问题
1-2句话说清楚：现在有什么bug/缺什么能力，不修会怎样。

## 改动思路
先说"为什么这么改"，再说"改哪些地方"。
每个改动点用一句话讲目的，再跟一句具体动作。
别把代码diff翻译成中文就当计划。

## 改动清单
按顺序列，每条格式：
- [文件/模块]：目的 → 具体动作
  先写中文功能描述，代码标识符放括号里。

## 边界
明确说不做什么，避免理解偏差。

## 验证
跑哪几条命令，断言什么结果。
```

### 计划模板对照

## 技术文档/Readme 模板

```
# 项目名
**一句话核心卖点：它能帮你解决什么核心痛点，一句话说清**

---

## 这是什么
一句话定义，再加一句大白话解释，核心要做的2-3件事，用短句列出来。

## 它和同类工具啥不一样
先说市面上同类产品的普遍短板，再说我们死磕的核心差异，每一点都讲实的，不玩虚的。

---

## 核心能力
### 已经跑通的功能
短句列，每一条都是真实落地的结果，不用修饰词。

### 正在补齐的内容
短句列，每一条都是明确的动作，不画饼。

---

## 当前进度
一句话说清核心进度，再列关键落地的细节，有凭有据。

## 先说清楚不做什么
明确边界，不绕弯，告诉用户当前不做的内容，避免误解。

---

## 技术栈
短句列，清晰明了。

---

## 快速跑起来
分模块给命令，每一步都有明确动作，不用多余解释。

---

## 建议先读的文档
按优先级列链接。

---

## 项目原则
短句列，每一条都是核心做事逻辑，不玩虚的。
```
